I'm building **Alex Photo Report** — a Telegram bot for verifying merchandising compliance at retail stores for AVTD (Торговий Дім АВ, Ukraine). The bot uses Claude Vision API to analyze shelf photos and generate reports.

**Stack:** FastAPI/Render, Neon PostgreSQL, python-telegram-bot, Claude Vision API

**Bot files:** `backend/photo_report/`

## ISSUE 1 (CRITICAL): False Shelf-Talker / Shelf-Stopper Detection

### Problem
The AI consistently reports: "Відсутній шелфстрипер GREENDAY при наявності конкурентських POS-матеріалів" (Missing GREENDAY shelf-strip when competitor POS materials are present). But **there are NO competitor shelf-talkers** in the photos. The AI mistakes **store price tags** (white paper labels with prices like "126,00", "75,00" etc.) for competitor POS materials.

### Evidence (from test reports)
- Report #12: Expert says "Ошибочно идентифицирует шелфтокер - их там нет"  
- Report #13: Same false detection of shelf-talkers  
- Report #14: "С шелфтокерами то же самое"  
- Report #15: "По шелфам то де самое"  
- Report #17: Same issue  
- Report #18: "Те же ошибки, и по посам, и по %"  

### Root Cause
The Vision model sees white horizontal paper strips under shelves (standard Ukrainian store **price tags** showing numbers like "126,00 грн") and misidentifies them as competitor branded POS materials (shelf-strips/shelf-talkers).

### Required Fix in SYSTEM_PROMPT
Add clear differentiation rules:

```
CRITICAL — POS Material Identification Rules:

A COMPETITOR SHELF-TALKER/SHELF-STRIP is ONLY:
- A plastic or cardboard strip attached to the shelf edge
- That displays a COMPETITOR BRAND LOGO or BRAND NAME (e.g., Nemiroff, Khortytsia, Absolut logo)
- Usually colored with brand colors, not white/plain

The following are NOT competitor POS materials:
- Store price tags (paper/plastic cards showing PRICES as numbers like "126,00", "75,00 грн") — these are standard retail price labels
- Shelf edge labels with product names and prices — these are store inventory labels
- White/plain paper strips under products showing numbers — these are price tags

RULE: Only flag missing AVTD shelf-strip if you see ACTUAL branded competitor materials with competitor LOGOS visible. Price tags with numbers ≠ competitor POS.

If unsure whether something is a competitor POS or a store price tag, DEFAULT TO "store price tag" — do NOT flag it as competitor POS.
```

## ISSUE 2 (CRITICAL): Vodka Shelf Share Calculation Error — Counting Only GreenDay Instead of ALL AVTD Brands

### Problem
The AI calculates "Доля полиці горілки" (vodka shelf share) by counting only GreenDay facings, but it should count ALL AVTD vodka brands together: **GreenDay + Ukrainka + Helsinki**.

### Evidence
- Report #12: Expert says "водку нашу он определил в 50% — там 3 водочных полки из 8и. То есть он считает долю ГД по наше продукции, а должен считать ГД+УКР+ХЕЛ от всей водки"
- Report #15: "Наш ГД посчитал меньше 25% — что неверно, коньяк и вино наше не видит вообще"

### Root Cause
The shelf share formula in the prompt likely only counts GreenDay facings as "ours" and treats Ukrainka and Helsinki as competitors.

### Required Fix in SYSTEM_PROMPT

```
AVTD VODKA BRANDS (all count as "OURS" for shelf share):
1. GREENDAY (Green Day) — green bottles/labels, variants: Classic, Crystal, Air, Lemon, Citrus
2. UKRAINKA (Українка) — clear/white bottles with "UKRAINKA" or "Українка" label
3. HELSINKI — blue/silver bottles with "HELSINKI" branding

VODKA SHELF SHARE FORMULA:
  our_share = (GreenDay_facings + Ukrainka_facings + Helsinki_facings) / total_vodka_facings × 100%

Where total_vodka_facings = ALL vodka bottles on the shelf (ours + competitors like Khortytsia, Nemiroff, Prime, Hlibny Dar, etc.)

CRITICAL ERROR TO AVOID: Do NOT count only GreenDay as "ours". ALL THREE brands (GreenDay, Ukrainka, Helsinki) are AVTD products.

Minimum threshold: ≥25% = PASS, <25% = FAIL
```

## ISSUE 4 (IMPORTANT): "General Overview" False Negatives — Not Recognizing Store Boundaries

### Problem
The AI flags "Відсутній загальний огляд всіх полиць" (Missing general overview of all shelves) even when the photo DOES show the full alcohol section. It doesn't recognize that refrigerators, end-caps, or category transitions mark the boundaries of the alcohol zone.

### Evidence
- Report #16: Expert says "Не определил края алкогольных полок как пограничные — там холодильники с 2х сторон как границы алкогольных полок"
- Report #17: "Да, общего вида нет. Доля 60%, он, я так понял, округляет"
- Photos 5, 6, 7, 10 clearly show full alcohol sections bounded by fridges and store corners

### Required Fix in SYSTEM_PROMPT

```
GENERAL OVERVIEW DETECTION:
A photo counts as a "general overview" if BOTH edges of the alcohol shelf zone are visible.

Boundaries/edges can be ANY of:
- Wall
- Refrigerator/cooler (glass door fridge)
- Corner of the room
- Different product category (snacks, non-food, etc.)
- End of shelving unit
- Store counter/checkout area
- Door/entrance

The photo does NOT need to show walls specifically. If you can see where the alcohol shelves START and END (left + right boundaries), that IS a general overview.

Common layout: alcohol shelves in an L-shape or straight line, bounded by fridges on one/both sides — this IS a general overview.
```

## ISSUE 9 (CRITICAL ARCHITECTURE): Multi-Photo Stitching — Combine Data Across All Photos Into One Report

### Problem
A single store visit produces 3-5 photos (left wall, center, right wall, general overview, close-ups). Currently the bot either analyzes only the first photo or analyzes each photo independently. This causes wildly inaccurate shelf share calculations — the AI only "sees" part of the store and reports partial data as if it were the whole picture.

### Evidence
- Report #17: Expert says cognac was counted from photo 2 mixed with vodka from photo 1
- Report #16: "Доля как 100% придумана" — AI saw only one section and assumed it was everything
- Reports consistently show different percentages depending on which single photo was analyzed

### Required Implementation

**Option A: Sequential multi-image analysis (recommended for MVP)**
Send ALL photos from one report to Claude Vision in a single API call as multiple images. The SYSTEM_PROMPT must instruct Claude to:
1. First scan ALL images to understand the full store layout
2. Identify which photos overlap (same shelves from different angles)
3. De-duplicate — don't count the same bottle twice if visible in multiple photos
4. Calculate shelf shares based on the COMBINED view of all photos
5. Only then generate the report

```python
# In vision.py — send all photos at once
messages = [
    {
        "role": "user",
        "content": [
            {"type": "image", "source": {"type": "base64", "media_type": "image/jpeg", "data": photo1_b64}},
            {"type": "image", "source": {"type": "base64", "media_type": "image/jpeg", "data": photo2_b64}},
            {"type": "image", "source": {"type": "base64", "media_type": "image/jpeg", "data": photo3_b64}},
            # ... up to 5 photos
            {"type": "text", "text": "Analyze ALL photos as one store visit. These are different angles of the same alcohol section."}
        ]
    }
]
```

**Add to SYSTEM_PROMPT:**
```
MULTI-PHOTO ANALYSIS RULES:
You will receive 1-5 photos from the SAME store visit. These show different angles/sections of the same alcohol shelf area.

Step 1: SCAN all photos first. Identify:
  - Which photos show the same shelves from different angles (overlap)
  - Which photos show different sections (left wall vs right wall vs center)
  - Which photo is the "general overview" (shows full shelf zone with boundaries)

Step 2: BUILD a mental map of the full store shelf layout by combining all photos.
  - Do NOT double-count bottles visible in multiple photos
  - If the same bottle/section appears in 2 photos, count it ONCE

Step 3: CALCULATE shelf shares based on the COMPLETE combined view.
  - Vodka share = all AVTD vodka across ALL photos / all vodka across ALL photos
  - Same for cognac, wine, sparkling

Step 4: GENERATE one unified report for the entire store visit.

If only 1 photo is provided, note in the report: "Аналіз на основі 1 фото — результати можуть бути неповними"
```

**Option B: Two-pass analysis (more robust, higher cost)**
Pass 1: Analyze each photo individually, get structured JSON with counts per brand
Pass 2: Merge JSONs in Python, de-duplicate overlapping sections, calculate final shares

## ISSUE 10 (CRITICAL ARCHITECTURE): Deterministic Scoring Engine — Move Scoring Out of the LLM

### Problem
Currently, scoring (score out of 100, PASS/FAIL status, error list) appears to be generated by Claude as part of the Vision analysis. This is unreliable because:
- Claude may "forget" auto-fail rules mid-response
- Claude rounds percentages inconsistently
- The same photo can get different scores on repeated analysis
- No way to update scoring rules without rewriting the entire prompt

### Required Implementation

**Split the flow into two stages:**

**Stage 1: Claude Vision → Structured Facts (JSON)**
Claude's ONLY job is to identify what's on the shelves and return structured data. No scoring, no pass/fail, no error messages.

```python
# vision.py — Claude returns ONLY raw data
VISION_PROMPT = """
Analyze the shelf photos and return ONLY a JSON object with these fields:
{
  "photos_analyzed": 3,
  "has_general_overview": true,
  
  "vodka": {
    "greenday_facings": 12,
    "greenday_skus": ["Classic 0.5L", "Crystal 0.7L", "Air 0.5L"],
    "ukrainka_facings": 5,
    "helsinki_facings": 3,
    "competitor_facings": 8,
    "competitor_brands": ["Nemiroff", "Khortytsia"],
    "total_vodka_facings": 28
  },
  
  "cognac": {
    "adjari_facings": 10,
    "adjari_skus": ["3★ 0.5L", "5★ 0.5L", "Cherry 0.5L"],
    "dovbush_facings": 0,
    "klinkov_facings": 0,
    "jean_jack_facings": 2,
    "competitor_facings": 0,
    "total_cognac_facings": 12
  },
  
  "wine": {
    "villa_ua_facings": 6,

```

**Stage 2: Python Scoring Engine → Deterministic Score**
A separate Python function takes the JSON and applies business rules:

```python
# scoring.py — deterministic, testable, no LLM involved
def calculate_score(vision_data: dict) -> dict:
    score = 100
    errors = []
    info = []
    auto_fail = False
    
    # --- Vodka shelf share ---
    v = vision_data["vodka"]
    our_vodka = v["greenday_facings"] + v["ukrainka_facings"] + v["helsinki_facings"]
    total_vodka = v["total_vodka_facings"]
    vodka_share = round(our_vodka / total_vodka * 100) if total_vodka > 0 else 0
    
    if vodka_share < 25:
        auto_fail = True
        errors.append({
            "code": "1_108608",
            "text": f"Частка полиці горілки менше 25% — провал звіту (загальне)",
            "severity": "critical"
        })
    
    # --- Cognac shelf share ---
    c = vision_data["cognac"]
    our_cognac = c["adjari_facings"] + c["dovbush_facings"] + c["klinkov_facings"] + c["jean_jack_facings"]
    total_cognac = c["total_cognac_facings"]
    cognac_share = round(our_cognac / total_cognac * 100) if total_cognac > 0 else 0
    
    # --- POS materials ---
    pos = vision_data["pos_materials"]
    if pos["competitor_shelf_talkers_present"] and not pos["avtd_shelf_strip_present"]:
        score -= 15
        errors.append({
            "code": "1_108106",
            "text": "Відсутній шелфстрипер AVTD при наявності конкурентських POS",
            "severity": "major"
        })
    
    # --- General overview ---
    if not vision_data.get("has_general_overview"):
        score -= 15
        errors.append({
            "code": "NP59_200009734",
            "text": "Відсутній загальний огляд всіх полиць",
            "severity": "major"
        })
    
    # --- Premium shelf ---
    ps = vision_data["premium_shelf"]
    if ps["top_shelf_visible"]:
        if not any([ps["gd_evolution_present"], ps["gd_planet_present"], ps["gd_discovery_present"]]):
            score -= 10
            errors.append({
                "text": "Evolution/Planet/Discovery на елітній полиці — відсутні",
                "severity": "minor"
            })
    else:
        score -= 5
        info.append("Верхня полиця не видна на фото")
    
    # --- Auto-fail override ---
    if auto_fail:
        score = min(score, 55)
        status = "НЕ ПРОЙДЕНО"
    else:
        status = "ПРОЙДЕНО" if score >= 60 else "НЕ ПРОЙДЕНО"
    
    return {
        "score": max(0, score),
        "status": status,
        "errors": errors,
        "info": info,
        "shelf_shares": {
            "vodka": vodka_share,
            "cognac": cognac_share,
            # ... wine, sparkling
        }
    }
```

**Benefits of this split:**
- Scoring rules can be changed WITHOUT touching the Claude prompt
- Same JSON can be re-scored when rules change (no re-analysis needed)
- Easy to write unit tests for scoring edge cases
- Claude focuses on what it's good at (visual recognition) instead of math
- Consistent scores — same data always produces same score

## ISSUE 11 (IMPORTANT ARCHITECTURE): Photo Storage & Audit Trail

### Problem
For dispute resolution and quality auditing, the system must store original photos linked to each report. Without this:
- Managers can't verify AI mistakes
- No way to re-analyze old reports with updated prompts
- No evidence trail for compliance audits

### Required Implementation

**Store photos directly in Neon PostgreSQL:**

Since we already use Neon PostgreSQL, store compressed photos as `bytea` or base64-encoded text directly in the database. Telegram photos are typically 100-200KB each, so 5 photos per report = ~1MB — well within Neon's capabilities.

```python
# photo_storage.py
import base64
from io import BytesIO
from PIL import Image

async def compress_and_store(report_id: int, photos: list[bytes], db) -> int:
    """Compress photos and store in Neon PostgreSQL"""
    stored = 0
    for i, photo_bytes in enumerate(photos):
        # Compress to reduce storage (optional, Telegram already compresses)
        img = Image.open(BytesIO(photo_bytes))
        buf = BytesIO()
        img.save(buf, format="JPEG", quality=80)
        compressed = buf.getvalue()
        
        await db.execute(
            """INSERT INTO report_photos (report_id, photo_order, photo_data, size_bytes)
               VALUES ($1, $2, $3, $4)""",
            report_id, i + 1, compressed, len(compressed)
        )
        stored += 1
    return stored

async def get_report_photos(report_id: int, db) -> list[bytes]:
    """Retrieve all photos for a report from Neon"""
    rows = await db.fetch(
        """SELECT photo_data FROM report_photos 
           WHERE report_id = $1 ORDER BY photo_order""",
        report_id
    )
    return [row["photo_data"] for row in rows]
```

**Flow update:**
1. Agent sends photos → bot receives them
2. Bot stores compressed photos to Neon PostgreSQL → gets report_photo IDs
3. Bot sends photos to Claude Vision → gets JSON
4. Bot runs scoring engine on JSON → gets score
5. Bot saves to DB: report + raw vision JSON + score (photos already stored in step 2)
6. Bot sends formatted report to agent

**Re-analysis capability:**

async def re_analyze_report(report_id: int, db):
    """Re-run analysis on stored photos with current prompt version"""
