# REPLIT AGENT — AUDIT ONLY (No Code Changes)
## Issue: Personal Telegram messages leaking into /hr Analytics Dashboard

---

## CONTEXT

The `/hr` admin dashboard at `gradus-ai.onrender.com/hr` shows a chat analytics table
(user messages, response times, types). The table is currently displaying messages from
**personal Telegram chats** of the bot owner's account — not just bot interactions.

This is a **privacy + security issue**. The audit must identify root cause(s) before
any fix is written.

---

## YOUR TASK: AUDIT ONLY — DO NOT MODIFY ANY FILES

Read the files listed below. Report findings. Do not write code, do not create migrations,
do not modify anything. This is a read-only investigation.

## STEP 1 — FIND THE CHAT LOG TABLE

Search the codebase for the table that stores chat messages shown in the dashboard.
Look for any of these names: `chat_logs`, `messages`, `bot_messages`, `maya_logs`, `user_messages`.

Report:
- Exact table name

- Is there a `chat_id` column? A `bot_source` or `source` column? A `user_id` column?

## STEP 2 — FIND THE TELETHON CLIENT AND ITS EVENT HANDLERS

The system uses Telethon with a **personal account session** (StringSession stored in
`TELETHON_SESSION` env var), NOT a bot token. This means Telethon receives ALL messages
from ALL chats the account is in — personal DMs, groups, channels, everything.

Find the Telethon client initialization and ALL registered event handlers.
Check these files first:
- `backend/services/hunt_tg_scraper.py`
- `backend/routes/telegram_webhook.py`
- Any file that imports from `telethon`

Report for EACH event handler found:
- File name and line number
- What event type it listens to (`NewMessage`, `CallbackQuery`, etc.)
- Does it filter by `chats=` parameter? (e.g. `@client.on(events.NewMessage(chats=[...]))`)
- Does it filter by `chat_id` inside the handler body?
- Does it call any logging/save function? What function name?

## STEP 7 — SAMPLE THE LEAKED DATA (if safe to do so)

Run this READ-ONLY query against the Neon DB and report results:

```sql
SELECT
    chat_id,
    bot_source,
    COUNT(*) as message_count,
    MIN(created_at) as first_seen,
    MAX(created_at) as last_seen
FROM chat_logs   -- replace with actual table name from Step 1
GROUP BY chat_id, bot_source
ORDER BY message_count DESC
LIMIT 20;
```

Report the output. This will show which chat_ids are in the table and confirm
which ones are personal chats vs known bot groups.
